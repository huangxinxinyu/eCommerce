# 1. 项目背景：
本系统是一个基于 Spring Cloud 的纯后端分布式架构，旨在模拟双 11 级别的瞬时大流量秒杀场景。系统通过 Nginx 与 Gateway 构建多层防护，利用 Redis + Lua 实现极速库存预扣，并配合 RocketMQ 异步解耦，确保在高并发下系统依然能保持最终一致性，实现高可用、高并发、高扩展性。

# 2. 核心技术栈 (Technology Stack)
接入层： Nginx 负责 L7 负载均衡，将流量均匀分发给 Gateway 节点。
控制层： Gateway 负责流量清洗（限流）与安全检查。
通信层： 内部微服务通过 Dubbo RPC 同步调用，关键路径采用 RocketMQ 异步解耦。
执行层： Redis + Lua 承载瞬时峰值，MySQL 保证最终数据落地。

维度	技术选型	说明
基础框架	Spring Boot 3.x + Spring Cloud Alibaba	微服务治理核心
RPC 框架	Apache Dubbo 3.x	取代 OpenFeign，提供高性能、低延迟的微服务调用
注册中心	Nacos	同时作为 Dubbo 服务发现与 Spring Cloud 配置中心
分流与均衡	Nginx + Gateway	Nginx 做最前端分流，Gateway 做业务侧治理
消息中间件	RocketMQ	事务消息、削峰填谷、延迟消息关单
持久层	MySQL + MyBatis-Plus	最终数据落地
本地治具	Docker Compose + Jmeter	用于本地分布式环境搭建与高并发压测

# 3. 系统架构设计 (Service Architecture)
3.1 服务拆分
1. Gateway Service (入口网关):
  - 职责：路由转发、滑动窗口限流、CORS 处理。
2. Store Service (秒杀核心):
  - 职责：热点商品信息缓存、Lua 脚本库存预扣、一人一单校验。
3. Order Service (订单):
  - 职责：异步创建订单、自动关单逻辑 (MQ 延迟消息)、状态同步。
4. Pay Service （银行服务）
  - 职责：管理支付流水、模拟银行交互、回调处理。
5. EmailService(模拟基础设施 不用真正做邮件或者短信服务):
  - 职责：通过 Log 打印模拟阿里云短信推送、邮件通知、支付接口回调。
3.2 流量链路 (Traffic Flow)
1. Nginx 层：接收公网请求，通过 upstream 负载均衡到两个 Gateway 实例。
2. Gateway 层：执行基于 Redis 的滑动窗口限流，非法请求直接拦截。
3. Service 层：Seckill 服务通过 Lua 脚本与 Redis 交互，扣减成功后发送消息至 MQ。
4. Async 层：Order 服务消费消息并落库。

# 4. 技术目标
## 1. Lua 脚本库存预扣
- 使用 Lua 脚本保证库存扣减的原子性，避免并发问题
- 预扣机制：先锁定库存，支付成功后真正扣减，超时自动回滚
- 脚本需支持：库存查询、预扣、回滚操作
## 2. 防超卖与一人一单
- 防超卖 ：Lua 脚本中校验库存数量，确保扣减后不小于 0
- 一人一单 ：基于用户 ID + 秒杀商品 ID 组合键，Redis Set 或数据库唯一索引约束
- 并发场景下需保证校验与扣减的原子性
## 3. 秒杀流程优化
- 请求拦截：Gateway 层提前过滤非法请求
- 热点数据缓存：商品信息、库存预热到 Redis
- 异步化：下单成功后异步落库，避免阻塞主流程
- 批量处理：MQ 消息批量消费，提升吞吐量
## 4. 未支付订单自动关闭
- 使用RocketMQ 延迟消息实现定时扫描
- 策略：订单创建时设置过期时间，扫描未支付订单并关闭
- 释放预扣库存：关单时回滚 Redis 预扣数量
## 5. 支付与关单并发控制
- 分布式锁：支付回调与定时关单操作需加锁，防止重复处理
- 状态机：订单状态流转需严格校验，防止状态不一致
- 幂等性：支付回调需支持幂等，防止重复扣款
## 6. 缓存优化
- 缓存击穿 ：热点 key 过期时，使用互斥锁或逻辑过期策略
- 热点缓存 ：商品信息使用本地缓存（Caffeine/Guava）+ Redis 二级缓存
- 缓存一致 ：旁路缓存和延迟双删
## 7. 数据一致性保证
- 最终一致性 ：通过 MQ 异步落库，补偿失败重试
- 本地消息表 ：记录操作状态，失败时补偿
## 8. 多级缓存架构
```
请求 → Nginx → Gateway → Local Cache → Redis 
→ MySQL
```
- L1 本地缓存 ：进程内缓存，热点数据预加载，过期时间短
- L2 分布式缓存 ：Redis 共享缓存，数据一致性高
- L3 数据库 ：最终数据落地
## 9. 滑动窗口限流
- 基于 Redis ZSet 实现滑动窗口统计请求频率
- 限流维度：用户 ID + 接口，支持细粒度控制
- 阈值可配置，支持动态调整
