## 2. 架构分层设计

### 2.1 接入层 (Nginx)

**职责**：
- 最前端的负载均衡入口
- 将流量均匀分发给多个 Gateway 实例
- 提供基础的故障转移能力

**配置要点**：
- upstream 配置多个 Gateway 节点
- 权重策略：支持动态调整
- 健康检查：定期探测节点存活状态

### 2.2 网关层 (Gateway Service)

**职责**：
- 路由转发：基于请求路径分发到后端服务
- 流量治理：滑动窗口限流
- 安全检查：CORS 处理、请求参数校验
- 请求聚合：减少前端请求次数

**技术实现**：
- Spring Cloud Gateway
- Redis ZSet 实现滑动窗口限流
- 过滤器链模式处理请求

### 2.3 服务层 (Business Services)

#### Store Service (秒杀核心)

**职责**：
- 秒杀商品管理
- 热点商品信息缓存
- **Lua 脚本库存预扣**（原子操作）
- **一人一单校验**
- 秒杀活动配置

**核心流程**：
```
用户下单请求
    ↓
校验秒杀活动状态
    ↓
检查一人一单（Redis Set）
    ↓
执行 Lua 脚本预扣库存
    ↓
发送订单创建消息（MQ）
    ↓
返回下单成功
```

#### Order Service (订单服务)

**职责**：
- 异步创建订单（MQ 消费）
- 订单状态管理
- **未支付订单自动关闭**（RocketMQ 延迟消息）
- 订单查询

**核心流程**：
```
MQ 消费订单创建消息
    ↓
落库创建订单（待支付状态）
    ↓
发送延迟关单消息（15分钟）
    ↓
MQ 消费关单消息
    ↓
查询订单状态
    ↓
未支付 → 回滚库存 → 更新订单状态为已关闭
    ↓
已支付 → 不做任何处理
```

#### Pay Service (支付服务)

**职责**：
- 支付流水管理
- 模拟银行交互
- **支付回调处理**
- **幂等性控制**

**核心流程**：
```
接收支付请求
    ↓
创建支付流水
    ↓
模拟银行处理
    ↓
回调通知 Order Service
    ↓
真正扣减库存（Confirm）
```

#### Email Service (通知服务)

**职责**：
- 订单创建成功通知（Log 打印模拟）
- 支付成功通知（Log 打印模拟）
- 短信推送模拟（Log 打印）

### 2.4 数据层

#### Redis 集群

**用途**：
- 库存预扣（Lua 脚本保证原子性）
- 一人一单校验（Set 结构）
- 滑动窗口限流（ZSet 结构）
- 会话缓存
- 热点数据缓存

**Key 设计**：
```
stock:预扣:{skuId}           # 预扣库存数量
user:order:{skuId}:{userId}  # 一人一单标记
rate:limit:{userId}:{api}    # 限流计数器
```

#### MySQL 集群

**表结构**：
- `seckill_spu`：秒杀商品表
- `seckill_stock`：秒杀库存表
- `order_info`：订单表
- `payment_info`：支付流水表

## 3. 服务间通信设计

### 3.1 同步通信 (Dubbo RPC)

**调用链路**：
```
Gateway → Store Service → Order Service
```

**配置要点**：
- 超时时间：500ms
- 重试次数：0（秒杀场景不重试）
- 负载均衡：权重随机

### 3.2 异步通信 (RocketMQ)

**消息队列设计**：

| Topic | 用途 | 消费者 |
|-------|------|--------|
| seckill-order | 订单创建消息 | Order Service |
| order-close | 延迟关单消息 | Order Service |
| order-pay-callback | 支付回调通知 | Order Service |
| stock-confirm | 库存确认消息 | Store Service |

**消息可靠性**：
- 事务消息保证库存和订单一致性
- 失败重试机制（最大3次）
- 死信队列处理

## 4. 高并发设计

### 4.1 多级缓存架构

```
请求 → Nginx → Gateway → Local Cache → Redis → MySQL
       (静态)   (进程内)  (分布式)    (最终)
```

**L1 本地缓存 (Caffeine)**：
- 热点商品信息
- 过期时间：30秒
- 最大容量：10000条

**L2 分布式缓存 (Redis)**：
- 库存预扣数据
- 一人一单标记
- 限流计数
- 过期时间：跟随秒杀活动

**缓存策略**：
- 旁路缓存（Cache-Aside）
- 延迟双删保证一致性
- 逻辑过期解决缓存击穿

### 4.2 库存预扣设计

**预扣流程 (TCC 思想)**：

```
Try（预留）
  Lua 脚本原子扣减库存
  设置用户购买标记
  
Confirm（确认）
  支付成功后真正扣减库存
  删除预扣标记
  
Cancel（回滚）
  超时/取消时回滚预扣库存
  删除购买标记
```

**Lua 脚本原子操作**：
```lua
-- 库存预扣脚本
local stock = redis.call('GET', KEYS[1])
if stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    redis.call('SADD', KEYS[2], ARGV[2])
    return 1
end
return 0
```

### 4.3 防超卖与一人一单

**防超卖**：
- Redis 原子扣减，Lua 脚本判断库存 ≥ 扣减数量
- 数据库唯一索引约束

**一人一单**：
- Redis Set 快速校验：`SISMEMBER`
- 数据库唯一索引兜底：`UNIQUE(user_id, sku_id)`

### 4.4 滑动窗口限流

**实现原理**：
```
ZSet 结构：
  Member: 请求 ID
  Score: 时间戳
  
滑动窗口算法：
  1. 删除窗口外的旧数据
  2. 添加当前请求
  3. 统计窗口内请求数
  4. 判断是否超过阈值
```

**配置参数**：
- 窗口大小：1分钟
- 阈值：100次/用户
- 限流维度：用户 ID + 接口路径

## 5. 订单生命周期

```
创建订单(待支付)
    ↓
支付中(用户跳转银行)
    ↓
┌──► 支付成功 ──► 完成
│
└──► 超时未支付 ──► 自动关闭(回滚库存)
    ↓
已取消
```

**状态流转**：
- `PENDING`：待支付
- `PAID`：已支付
- `CLOSED`：已关闭
- `CANCELLED`：已取消

## 7. 容灾与降级

### 7.1 限流降级

- 网关层全局限流
- 服务级别限流
- 触发条件：QPS 超过阈值

### 7.2 熔断机制

- Sentinel 实现熔断降级
- 慢调用比例、异常比例触发
- 恢复时间：30秒

### 7.3 故障转移

- Nginx upstream 自动摘除故障节点
- Dubbo 负载均衡自动跳过不健康实例
